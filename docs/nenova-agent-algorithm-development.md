# Nenova Agent Algorithm Development Contract

Date: 2026-05-25 KST

## Purpose

Nenova/Orbit is not a static dashboard. Every layer must improve the agent
algorithm over time. The product should continuously turn `nenova.exe` usage,
KakaoTalk/KakaoWork uploads, and PC activity data into verified work units,
assistance actions, and product development tasks.

This contract applies to every Nenova work-flow feature, screen, API, agent,
and automation.

## Core Loop

```text
collect evidence
  -> normalize events
  -> create work-unit candidates
  -> cross-validate sources
  -> decide confidence and risk
  -> assist employee or manager
  -> record outcome
  -> improve algorithm, UI, API, and sales proof
```

No feature is complete unless it leaves data that can improve the next loop.

## Evidence Families

Every operational claim should try to use at least two evidence families:

| Evidence family | Examples | Role |
| --- | --- | --- |
| Conversation | KakaoTalk, KakaoWork, room, sender, message intent | Why the work happened |
| PC activity | app, window, keyboard, mouse, screen summary, file/process context | What the employee actually did |
| Nenova work | `nenova.exe`, ERP/customer/order/product/session evidence | Business-system confirmation |
| Outcome | follow-up message, schedule, completion, correction, manager review | Whether the interpretation helped |

Single-source claims must be marked as `검증대기`.

## Agent Roles

| Agent | Algorithm responsibility |
| --- | --- |
| `nenova-data-fusion` | Build normalized work-unit candidates from raw evidence |
| `nenova-workflow-forecaster` | Predict next work, bottlenecks, schedule risks, and automation candidates |
| `nenova-cross-validator` | Attack the claim, find conflicts, and assign PASS/WARN/FAIL |
| `nenova-ops-orchestrator` | Convert verified findings into employee help, manager help, and product tasks |
| Codex | Implement the smallest validated product improvement |
| Claude | Independently challenge Codex's interpretation through the agent team |

## Required Algorithm Fields

Any new endpoint, UI panel, report, automation, or persisted event that touches
employee/company work should preserve these fields when possible:

| Field | Meaning |
| --- | --- |
| `employeeId` / `employeeName` | Who did the work |
| `startedAt` / `endedAt` / `observedAt` | When the work happened |
| `sourceFamilies` | Which evidence families support the claim |
| `workUnitId` | Stable unit for later review and learning |
| `intent` / `taskType` | What the work appears to be |
| `businessContext` | Customer, product, order, project, room, or account |
| `confidence` | 0-100 confidence score |
| `validationStatus` | `일치`, `부분일치`, `충돌`, `검증대기` |
| `conflicts` | Source disagreements and missing evidence |
| `nextAction` | Employee/manager/automation action suggested by the agents |
| `outcome` | Accepted, ignored, corrected, completed, failed, or unknown |
| `algorithmLearning` | What the system learned or should change next |

## Development Rules

1. Every feature must ask: "How does this improve the next interpretation?"
2. Every Work Unit must be explainable by evidence, not only by LLM text.
3. Every agent output must include confidence and missing evidence.
4. Every UI decision should expose the reason behind the recommendation.
5. Every automation candidate must keep before/after proof.
6. Every product improvement should be traceable to a validation packet or data gap.
7. If data is unavailable because the PC is offline, the system must record that as a resilience gap, not silently fail.

## Codex-Claude Mutual Validation

The standard validation artifact is generated by:

```bash
node scripts/nenova-cross-agent-validation.js
```

When Claude CLI is authenticated, run:

```bash
node scripts/nenova-cross-agent-validation.js --claude
```

The validation packet should become the source of truth for the next development
task. Codex should only implement one small improvement per run unless the user
explicitly asks for a larger refactor.

## Sellable Product Requirement

The algorithm development loop must produce evidence that can be sold:

- employee work understanding accuracy,
- company process discovery,
- source coverage and validation rate,
- unresolved data gaps,
- automation candidates with estimated impact,
- assistance actions accepted or corrected by employees,
- manager summaries that reduce manual supervision,
- cloud/offline resilience proof.

The product is not "AI lectures" or "tool usage." It is a continuously improving
Korean SME work OS that proves what work happened, why it happened, and what the
company should do next.
