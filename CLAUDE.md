## 절대 규칙

### 0. 반복 감지 진단 프로토콜 (최우선)
**사용자가 이전에 했던 지시와 겹치는 요청을 하면, 절대로 바로 다시 구현하지 않는다.**

대신 다음 진단 프로세스를 따른다:

```
Step 1: 커밋 확인
  → git log --oneline -20 에서 관련 커밋이 있는지 확인
  → 있으면 "이 작업은 커밋 XXXX에서 이미 했습니다" 보고

Step 2: 코드 존재 확인
  → 관련 파일에 코드가 실제로 있는지 grep/read
  → 있으면 "코드는 존재합니다" 보고

Step 3: 왜 안 되는지 원인 진단
  → 3가지 중 하나:
  (A) 코드는 있지만 프로덕션 미배포 → git push 필요
  (B) 코드는 있지만 UI 흐름 연결 누락 → 연결 코드만 추가
  (C) 코드는 있지만 다른 버그로 안 됨 → 버그 특정 후 수정

Step 4: 최소 수정
  → 전체를 다시 만들지 않고, 원인에 맞는 최소 수정만 한다
  → 수정 후 프로덕션 검증까지 완료

Step 5: 메모리 업데이트
  → 문제 원인과 해결 방법을 MEMORY.md 또는 bugfixes.md에 기록
  → 같은 문제가 다시 발생하지 않도록 방지
```

**보고 형식 (매번 이 형식으로):**
```
🔍 반복 감지: [요청 내용]
📋 이전 작업: 커밋 XXXX (날짜)
✅ 코드 존재: [파일명]
❌ 미동작 원인: [A/B/C 중 하나 + 상세]
🔧 최소 수정: [수정 내용]
```

### 1. 작업 반복 금지
- 위 진단 프로토콜을 반드시 먼저 실행
- 이미 구현된 기능을 다시 만들지 않는다
- 수정이 필요하면 기존 코드를 **편집**한다 (새로 작성 X)

### 2. 웹 서비스 기획 관점 유지
- 모든 답변은 "Orbit AI 웹 서비스"의 기획/개발 맥락이다
- 기능 추가 시 사용자 경험(UX)과 비즈니스 가치를 함께 고려
- 단순 코딩이 아닌, 제품으로서의 완성도를 추구

### 3. 프로덕션 우선
- 로컬(SQLite)과 프로덕션(PostgreSQL) 동시 호환 필수
- `src/db.js` 수정 시 반드시 `src/db-pg.js`도 동기화
- 배포: GitHub main push → Railway 자동배포

## 제품 철학 (절대 원칙)

### 데이터 격리
- 각 계정은 자기 데이터만 본다. 다른 사람 데이터 절대 노출 금지.

### 노드 텍스트 = 분석 결과
- 사용자가 작업한 원본 내용을 그대로 보여주는 것이 **아니다**
- 수집된 데이터를 **분석**한 뒤, 이 사람이 **어떤 프로젝트**, **어떤 목적**을 위해 작업하는지를 나타낸다
- 예: "git push" ❌ → "Orbit 플랫폼 — 인증 시스템 구축" ✅

### 데이터 수집 목적
- 개인 업무를 모두 데이터화하여:
  1. **업무 효율** 향상 방법 발견
  2. **루틴** 확인 및 **트리거** 감지
  3. **AI 자동화** 가능 영역 발견

### 7대 핵심 가치

| # | 기능 | 설명 |
|---|------|------|
| 1 | **관리자 대시보드** | 회사 업무 프로세스 한눈에 확인, 직원별 진행도·효율 모니터링 |
| 2 | **역량 레퍼런스** | 사용자의 작업 분석 → LinkedIn처럼 개인 역량 포트폴리오화 |

| 4 | **다중작업 미리보기** | 현재 진행 중인 여러 작업을 미리보기로 한눈에 확인 |

### 에이전트 아키텍처
- 모든 기능을 **에이전트화**하여 기능 충돌 없이 독립 동작
- 질문/답변은 **깃 + 드라이브에 자동 저장**
- 사용자의 **학습 데이터를 볼 수 있는 별도 시트** 구성
